1. Introduction: Getting to Know the Shell

Before we go into the details of the bash shell, we need to clarify a few concepts that are important to a solid foundation. Although not with all the details, having an idea of how the system we will use works in the background and what kind of structure it has, enables us to perform all the operations we will do in the future in a much more conscious way.

1.1 Operational Overview

Since you want to learn bash shell programming, you know that Linux is actually the kernel, and the GNU formation provides various tools, and the operating system (GNU / Linux) as a result comes into being. You may have even heard what the kernel does. Now, as a short reminder, let's try to make sense of what is happening by looking a little closer to this process.

We can abstract our computer (hardware) and the operating system (software) running in it in the simplest form as follows.

Since we are not yet going to cover which hardware has which functionality, let's start by looking directly at the "operating system" section.

1.1.1 Operating System

The operating system is an intermediary abstraction layer, in other words a software interface, that allows us to communicate with the hardware (disk, ram, cpu, monitor, network cards, etc.) in our computer. Thanks to the operating systems, we can easily work on existing hardware without thinking about how to communicate with each hardware separately. If this were not the case, how could we maintain thousands of kinds of brand and model equipment together?

We said that the operating system saves us from dealing with hardware. But how exactly does it do this?

The operating system basically consists of two parts. These partitions are called "user space" and "kernel space". As users who manage the system, we use tools in the user area. These tools also communicate with the kernel. The kernel, on the other hand, transmits the orders from the user area to the hardware and ensures that the relevant work is done.

Note: The diagram below includes representative tools in the user area, so of course not all system tools.

1.1.1.1 User Space

As a standard user we would like to deal with very little while doing any work on the system. For example, when I am writing on a text document, I do not have to think about how to get the hardware done, it shouldn't. I just use my keyboard and write and leave the rest to the operating system. The operating system organizes all the necessary operations thanks to its layered structure, as we explained under the previous heading.

Since we are users in the entire structure, the "user area" has been designed in such a way that we can easily manage the system in line with our wishes . In other words, we can call the user area the command control area. Moreover, the user area offers us two alternative environments for system management. These are "graphical user interface (GUI)" and "command line interface (CLI)" environments.

1.1.1.1.1 Graphical User Interface

What we mean by the graphical interface is the working environment where visuality is at the forefront, where we can manage the system with a set of clicks and drag and drop operations, usually using the mouse. In this environment, there are windows, buttons, drag and drop, you know, I am talking about the graphical user interface that we always come across. When it comes to the graphical interface environment, our display server, desktop environment and window manager are toolsets that make it easy for us to use the graphical interface. We can only use the graphical interface tools installed in our system on the graphical interface environment. We can give examples of graphic interface tools to office software.

1.1.1.1.2 Command Line Interface

The command line interface is the environment in which tools that do not have a graphical interface are run. In this environment, you cannot use tools that work with a graphical interface because this environment is not designed to provide a graphical interface. Tools that run on the command line interface are tools that can be managed on the keyboard without the need for graphical interaction. Since there is no graphical interaction, the shell helps us for tasks such as finding and running tools, switching between tools or common use. Just as applications with a graphical interface are easy to manage thanks to peripherals such as a desktop environment, it provides us with the environment we need to manage the tools run via command line in the shell in an organized manner. In short, the structure we call "shell" is our aide on the command line.

We click on the application icon in the graphical interface, in the command line interface, we write the name of the relevant tool in the shell. Both environments give us the tools on the system; They help with "management" in subjects such as finding, operating, tracking and terminating when necessary.

Now let's look at the general operation of system management through a very simple example. For example, if I want to create a new folder, I have two basic options. The first is to use the file manager with a graphical interface, and the other is to create the folder by giving a command to the system management tool, which is called "shell".

Just right click on the graphical interface and say "create new folder". When we use the file manager tool, which has a graphical interface, this tool communicates with the kernel and expresses the process we want to perform to the kernel. The kernel explains how to do this to the hardware connected to the system, if all the conditions required for this process are suitable. Finally, our folder is created by running the necessary hardware. You can see the simple abstraction of this process flow in the diagram below.

Tools with a command line interface that do not have a graphical interface actually have a completely similar operation. To confirm this, let's take a look at the flow of creating a file with the command line interface. In order to order the action to be made to the system, I open my terminal tool and enter the required expression into the command line.

Here, the tool that allows us to create folders without the need for a graphical interface is the "mkdir" tool. We use the tool called "console" or "terminal" that reads the input on our keyboard to indicate to the system that we want to use this tool. The sole task of the console tool here is to take the inputs on our keyboard and transfer them to the "shell" structure. The structure called the shell makes sense of the command we have expressed in written form, finds the "mkdir" tool and enables it to be run with the "folder" argument. In this way, the "mkdir" tool that is run can explain to the kernel that it wants to create a folder named "folder". The kernel can do the work required for the hardware. As a result, we are able to easily perform the transaction we want with only written orders.

The most general operating structure of the system is as we have discussed. Finally, we can talk about a few more details and focus on the shell.

Before continuing, let's clarify all the concepts by talking about the "command line" tools known as "console" or "terminal", which are very confused with "shell".

1.1.1.1.3 Terminal | Console | Command Line

We mentioned that the shell allows us to manage the system without the need for a graphical interface. Since the shell does not have a graphical interface, we need to deliver our orders via the command line tools called "terminal" or "console" when communicating with the shell. Namely, command line tools are only obliged to take the input on our keyboard line by line to the shell and display the results returned by the shell. Command line tools do not interpret the given orders, it is the shell that interprets them.

To mention a little more in detail, there are virtual consoles on the system called TTY. Ctrl + Alt + F1 Ctrl + Alt + F2Other Ctrl + Alt + F2, F3, F4, F5, F6, F7keys can be used to switch between these consoles or depending on the situation .

If the graphical interface environment is installed in the current system, at least one of the TTY consoles is reserved for this graphical interface environment. Usually the 1st or the 7th TTY console is reserved to operate the graphical interface environment, although it varies depending on the distribution used. When you open any terminal tool that exists in the graphical interface environment, this terminal tool actually offers a command line, expressed as "so-called TTY (known as pseudo-terminal (PTY))", which enables us to issue orders to the shell even through the graphical interface. Thanks to all this structure, whether we are in a graphical interface environment or we only work in the command line interface, we can communicate with the shell thanks to the console tools.

While giving our folder creation example, we communicated with the shell using the terminal tool on its graphical interface, namely "pseudo-TTY (pseudo-terminal (PTY))". If we wish, we can repeat the same example without any problems by switching to the command line interface.

In the light of all these explanations, we have personally confirmed that it does not matter which "console" vehicle is and that the main thing to understand from our state is the shell.

The structure we call a shell is ultimately a software. Therefore, as with any software, the shell has many alternatives. In this tutorial, we will be focusing on the "Bash" shell, which is the most widely used in the world age.

1.2.2 Structure of Scripting Programming

The bash shell is actually both a command interpreter and a programming language. The command interpreter functionality provides the interface between the tools on the system and the user. Thanks to the shell, we can find these tools, run them and terminate them when necessary. Being a programming language allows the tools on the system to be combined and used in an organized manner when necessary. When the commands are collected properly in a single file, it is possible to manage many tools with a single command from a single file. In short, with shell programming, we have versatile system management tools that can easily provide solutions that fit our exact needs. Shells already offer us the possibility to program existing tools so that they can be operated together effectively.

1.2.3 Bash Version Used and Tracking Innovations

Of course, there will be innovations in bash shell over time. So the version discussed here may be out of date by the time you were reading it. You may even be deliberately using an older bash shell than the one covered in this book. In this case, is the information in this book null and void, except for the (bash v5.1) version in which it is discussed?

No way. Differences between versions are usually in the form of fixing existing bugs or introducing new alternative solutions (features and tools). Updates avoid replacing the old version builds because scripts that have already been written can potentially fail in new versions. If for some reason an existing vehicle or feature needs to be changed in a way that affects its basic operation, a new tool or feature containing the change is offered instead of replacing the existing one. In this way, while old features and tools can be used in new versions, new tools and features that can meet new needs are also offered. Ultimately, the changes on the shell are made to offer new solutions without destroying the existing one. Shell updates are backward compatible.

For these reasons, regardless of which version of bash you are using, it is sufficient to take a brief look at the "change notes" to see the differences between versions. When you browse through the change notes, you can often confirm that existing features have been improved or alternatively the same facilitating features have been introduced. I do not provide a direct link for you to view the exchange notes because this address may be invalid at the time you are reading. With just a short internet search, you can easily access a detailed list of all changes between versions.

1.2.4.2 Every Vehicle on the System Can Be Used

The main purpose of shell programming is to act as an adhesive between tools. It allows us to run the tools installed in the system together in an organized manner. Shell alone doesn't make sense anyway. What makes the use of a shell powerful is the tools installed on the system. The shell only provides the environment required for these tools to be run and their output to be transferred to other tools and reworked. You can think of the shell as a conductor. The sounds come from the instruments played by the musicians, but it is the conductor that makes all the musicians play in harmony.

In an example scenario; I can say to my script file "first run the program file created in python language, run the program created in perl language in line with the outputs of the file and print the last output to a file".

Since the process we do is to collect system management commands in a single file, we can ensure that tools developed with different programming languages work on a common ground, as we can do while managing the system. Thus, we can produce solutions to complex problems by using many different tools that perform basic functions together. Already, due to the unix philosophy, the tools on the system are usually prepared to do a single job in the best way. The combination of tools that do their job best can be operated in an organized manner, offering an unlimited variety of solutions. It is this approach that makes the use of the shell powerful.

In addition, it is also possible to obtain the "joint operation of tools" opportunity offered by the shell with different programming languages. However, in many cases the development phase will be long and demanding and the flexibility of the tools will be low compared to the shell.
